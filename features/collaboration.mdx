Brainbox is built on a **local-first** architecture that enables real-time collaboration, offline editing, and automatic conflict resolution using CRDTs (Conflict-free Replicated Data Types).

## How It Works

Brainbox's collaboration system operates on four core principles:

<Steps>
  <Step title="Local-first storage">
    Every client maintains a full SQLite database. All changes write locally first, ensuring instant responsiveness and offline capability.
  </Step>
  
  <Step title="Background sync">
    When connected, changes automatically sync to the server via WebSocket. The server persists data to PostgreSQL as the source of truth.
  </Step>
  
  <Step title="CRDT merge">
    Concurrent edits from multiple users merge automatically using Yjs CRDTs. No manual conflict resolution required.
  </Step>
  
  <Step title="Real-time updates">
    All connected clients receive live updates through WebSocket broadcasts, showing changes as they happen.
  </Step>
</Steps>

### CRDT-Based Merging

Brainbox uses **Yjs** for conflict-free collaborative editing:

<Tabs>
  <Tab title="How CRDTs Work">
    CRDTs (Conflict-free Replicated Data Types) are data structures that can be updated independently on multiple replicas and automatically merged:
    
    **Key properties:**
    - **Commutative** - Order of operations doesn't matter
    - **Associative** - Grouping of operations doesn't matter
    - **Idempotent** - Applying an operation twice has the same effect as once
    
    **Example:**
    
    ```typescript
    // User A inserts "Hello" at position 0
    // User B inserts "World" at position 0
    // Traditional merge: conflict!
    // CRDT merge: "HelloWorld" or "WorldHello" (deterministic)
    
    // With Yjs:
    import * as Y from 'yjs'
    
    const doc = new Y.Doc()
    const yText = doc.getText('content')
    
    // User A's edit
    yText.insert(0, 'Hello')
    
    // User B's edit (concurrent)
    yText.insert(0, 'World')
    
    // Result: Automatic merge with preserved intent
    ```
  </Tab>
  
  <Tab title="Rich Text Handling">
    Yjs handles rich text with formatting marks:
    
    - **Character-level tracking** - Each character has a unique ID
    - **Format preservation** - Bold, italic, etc. merge correctly
    - **Block structure** - Headings, lists, and blocks maintain integrity
    - **Deletion tracking** - Tombstones prevent resurrection of deleted content
    
    **Under the hood:**
    
    ```typescript
    // Brainbox page content structure
    {
      type: "rich_text",
      blocks: {
        "block_1": {
          id: "block_1",
          type: "paragraph",
          content: [
            {
              type: "text",
              text: "Collaborative editing",
              marks: [
                { type: "bold", attrs: {} }
              ]
            }
          ],
          index: "a0",
          parentId: null
        }
      }
    }
    ```
  </Tab>
  
  <Tab title="Database Merging">
    Database fields and records also use CRDTs:
    
    - **Field values** - Last-write-wins with vector clocks
    - **Field definitions** - Schema changes merge safely
    - **Record order** - Fractional indexing maintains sort order
    - **Views** - Filter and sort configurations sync independently
    
    **Conflict resolution:**
    
    | Scenario | Resolution |
    |----------|------------|
    | Same field edited by two users | Last write wins (by timestamp) |
    | Different fields edited | Both changes apply |
    | Record created with same ID | Server-assigned ID prevents collision |
    | View filters changed | Per-user view state (not synced) |
  </Tab>
</Tabs>

### Offline Workflow

<Steps>
  <Step title="Connection lost">
    WebSocket disconnects. UI shows offline indicator.
  </Step>
  
  <Step title="Continue working">
    All changes write to local SQLite database. No interruption to user experience.
  </Step>
  
  <Step title="Queue sync">
    Changes queue with pending sync status.
  </Step>
  
  <Step title="Reconnect">
    WebSocket reconnects automatically when internet returns.
  </Step>
  
  <Step title="Sync queued changes">
    Pending changes upload to server in order.
  </Step>
  
  <Step title="Merge conflicts">
    CRDTs merge any concurrent changes from other users.
  </Step>
  
  <Step title="Broadcast updates">
    Server broadcasts merged state to all clients.
  </Step>
</Steps>

### Handling Conflicts

<Info>
  In most cases, conflicts resolve automatically via CRDTs. Manual intervention is rarely needed.
</Info>

When conflicts do occur:

<Tabs>
  <Tab title="Automatic Resolution">
    Most conflicts resolve without user action:
    
    **Text edits:**
    - Concurrent insertions: Both apply, merged by CRDT
    - Concurrent deletions: Deleted content removed
    - Format conflicts: Both formats apply (text can be bold AND italic)
    
    **Database edits:**
    - Different fields: Both changes apply
    - Same field: Last write wins (by timestamp)
    - Record deletion: Tombstone propagates, record removed for all users
  </Tab>
  
  <Tab title="Edge Cases">
    Rare scenarios requiring special handling:
    
    **Schema changes:**
    - Adding field: New field appears for all users
    - Removing field: Field removed, data preserved in tombstone
    - Field type change: Migration applies with validation
    
    **File conflicts:**
    - Same filename: Automatic rename with timestamp
    - Concurrent uploads: Both files saved with unique IDs
    - Delete vs edit: Delete wins (file removed)
    
    **Permission conflicts:**
    - Role changes: Most recent role assignment wins
    - Access revoked: User sees read-only or loses access on next sync
  </Tab>
</Tabs>

## Sync Engine Implementation

The sync engine coordinates data flow between client and server:

### Client-Side Handlers

Located in `packages/client/src/handlers/`:

```typescript
// Example: Node updates handler
export async function handleNodeUpdates(
  input: NodesUpdatesInput
): Promise<void> {
  const db = await openDatabase();
  
  for (const node of input.data) {
    await db
      .insertInto('nodes')
      .values({
        id: node.id,
        type: node.type,
        attributes: JSON.stringify(node.attributes),
        updatedAt: node.updatedAt,
      })
      .onConflict((oc) =>
        oc.column('id').doUpdateSet({
          attributes: JSON.stringify(node.attributes),
          updatedAt: node.updatedAt,
        })
      )
      .execute();
  }
}
```

### Server-Side Broadcasters

Located in `apps/server/src/synchronizers/`:

```typescript
// Example: Broadcasting node updates
export async function broadcastNodeUpdate(
  nodeId: string,
  workspaceId: string
): Promise<void> {
  const node = await getNode(nodeId);
  
  await broadcastToWorkspace(workspaceId, {
    type: 'nodes-updates',
    data: [node],
  });
}
```

    - Chat messages

    - Permission changes

## Data Consistency

Brainbox ensures data consistency across clients:

### Consistency Guarantees

<AccordionGroup>
  <Accordion title="Eventual Consistency">
    All clients eventually converge to the same state:
    
    - **Guarantee:** Given no new updates, all replicas converge
    - **Timeline:** Typically within seconds of reconnection
    - **Verification:** Vector clocks track causal ordering
  </Accordion>
  
  <Accordion title="Causal Consistency">
    Operations preserve cause-and-effect relationships:
    
    - **Guarantee:** If operation A causes operation B, all clients see A before B
    - **Implementation:** Logical timestamps and dependency tracking
    - **Example:** Reply to message always appears after original message
  </Accordion>
  
  <Accordion title="Strong Consistency (Server)">
    Server database maintains strong consistency:
    
    - **Guarantee:** Server always has authoritative state
    - **Implementation:** PostgreSQL ACID transactions
    - **Use:** Conflict resolution and audit trail
  </Accordion>
</AccordionGroup>

### Conflict-Free Operations

Certain operations are guaranteed conflict-free:

| Operation | Conflict-Free? | Reason |
|-----------|----------------|--------|
| **Insert text** | ✓ | CRDT positions are unique |
| **Delete text** | ✓ | Tombstones prevent resurrection |
| **Create node** | ✓ | UUIDs prevent ID collision |
| **Update different fields** | ✓ | Independent field state |
| **Add reaction** | ✓ | Set-based CRDT |
| **Update same field** | ✗ | Last-write-wins |
| **Reorder items** | ~ | Fractional indexing reduces conflicts |

## Best Practices

<AccordionGroup>
  <Accordion title="Trust the sync engine">
    Don't manually refresh to see changes. The sync engine automatically propagates updates in real-time.
  </Accordion>
  
  <Accordion title="Work offline confidently">
    Brainbox is designed for offline work. Don't wait for internet connection - keep working and changes will sync later.
  </Accordion>
  
  <Accordion title="Avoid rapid field type changes">
    Changing field types (e.g., text to number) can cause data loss if users have offline edits. Plan schema changes carefully.
  </Accordion>
  
  <Accordion title="Monitor sync status">
    Watch the sync indicator in the UI. If changes aren't syncing, check your connection.
  </Accordion>
</AccordionGroup>

## Troubleshooting

<Tabs>
  <Tab title="Sync Issues">
    **Changes not syncing:**
    
    1. Check internet connection
    2. Verify WebSocket status in network tab
    3. Check browser console for errors
    4. Try disconnecting and reconnecting
    5. Clear local cache and re-sync
    
    **Slow sync:**
    
    1. Check network speed
    2. Reduce concurrent edits
    3. Limit file upload size
    4. Check server load
  </Tab>
  
  <Tab title="Conflict Issues">
    **Unexpected data loss:**
    
    1. Check server logs for conflict resolution
    2. Verify timestamps on conflicting changes
    3. Review CRDT merge logs
    4. Contact admin for data recovery
    
    **Duplicate content:**
    
    1. Likely from double-submit (network issue)
    2. Delete duplicate manually
    3. Report bug if reproducible
  </Tab>
  
  <Tab title="Offline Issues">
    **Can't work offline:**
    
    1. Ensure content was loaded while online
    2. Check browser storage quota
    3. Verify service worker is registered
    4. Clear cache and reload while online
    
    **Offline changes lost:**
    
    1. Check browser didn't clear storage
    2. Verify sync queue in developer tools
    3. Check for browser crashes during offline work
  </Tab>
</Tabs>
